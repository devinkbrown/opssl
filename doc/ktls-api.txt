opssl KERNEL TLS API REFERENCE
===============================

This document describes the Linux kernel TLS (kTLS) API in opssl/ktls.h.

After a successful TLS handshake, opssl can promote the connection to
kernel TLS. The kernel handles encryption/decryption transparently, and
the socket FD can be transferred to another process via SCM_RIGHTS for
zero-downtime live migration.

Prerequisite: the tls kernel module must be loaded (sudo modprobe tls
or CONFIG_TLS=y in the kernel config).

Supported kTLS ciphers: AES-GCM-128, AES-GCM-256, ChaCha20-Poly1305.


AVAILABILITY
============

    bool opssl_ktls_available(void);

Check if kTLS is available at runtime. Returns true if the tls kernel
module is loaded and the necessary setsockopt calls are supported.


PROMOTION
=========

    int opssl_ktls_promote(opssl_conn_t *conn);

Promote a completed TLS connection to kernel TLS. Must be called
immediately after handshake, before any application data is exchanged.
Flushes any pending post-handshake records first.

Returns:
     1 — promoted successfully (both TX and RX offloaded)
     0 — not promoted (cipher not supported by kernel, or module not loaded)
    -1 — partial promotion (FATAL: TX active but RX failed — must close)

A return of -1 is a fatal error: the connection is in an inconsistent
state where TX is kernel-managed but RX is not. The only safe action is
to close the connection.

    int opssl_ktls_promote_late(opssl_conn_t *conn);

Late promotion for established connections that were not promoted at
handshake time. Reads current sequence numbers from the live session.
Use before live migration for connections that have already exchanged
application data.


STATUS
======

    bool opssl_ktls_is_active(const opssl_conn_t *conn);
    bool opssl_ktls_tx_active(const opssl_conn_t *conn);
    bool opssl_ktls_rx_active(const opssl_conn_t *conn);

Check kTLS status. is_active returns true if both TX and RX are
offloaded. tx_active/rx_active check each direction independently.


I/O
===

    ssize_t opssl_ktls_read(opssl_conn_t *conn, void *buf, size_t len);
    ssize_t opssl_ktls_write(opssl_conn_t *conn, const void *buf, size_t len);

After kTLS promotion, these replace opssl_read/opssl_write. They are
thin wrappers around recv(2)/send(2) that handle:
- MSG_NOSIGNAL for writes (no SIGPIPE)
- close_notify detection via TLS_GET_RECORD_TYPE
- Proper errno propagation

In practice, ophion calls opssl_read/opssl_write which internally
dispatch to kTLS I/O when the connection is promoted.


FD ADOPTION
===========

    opssl_conn_t *opssl_ktls_adopt(int fd, opssl_direction_t dir);

Mark an adopted FD as kTLS-active. Called in the new process after
receiving a migrated kTLS socket via SCM_RIGHTS.

The returned connection object has no TLS library state — only the fd
and the kTLS flag. All subsequent I/O goes through the kernel. This is
the most efficient migration path: no key material crosses process
boundaries, no re-handshake needed.


KEY EXTRACTION
==============

    typedef struct {
        uint8_t  key[32];
        uint8_t  iv[12];
        uint8_t  rec_seq[8];
        size_t   key_len;
        uint16_t cipher_type;
        uint16_t tls_version;
    } opssl_ktls_keys_t;

    int opssl_ktls_extract_keys(opssl_conn_t *conn,
                                opssl_ktls_keys_t *tx,
                                opssl_ktls_keys_t *rx);

Extract raw key material for manual kTLS setup. Populates the TX and
RX key structures with the current traffic keys, IVs, and record
sequence numbers. cipher_type maps to TLS_CIPHER_AES_GCM_128 etc.
from <linux/tls.h>.

This is used internally by opssl_ktls_promote. Application code
typically does not need this unless implementing custom kTLS setup.
